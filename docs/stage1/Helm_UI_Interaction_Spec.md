## Purpose

This document defines the complete contract between the Helm UI (React frontend) and the Helm Runtime (FastAPI backend). Every API call, every SSE event, every response format, and every directive is specified here. Helm IDE builds the runtime against this contract. The UI team builds the frontend against this contract. When they meet at integration (Phase 3), both sides should fit together without negotiation.

Three layers:

1. **Layer 1 ‚Äî Request/Response Contract** (how the UI talks to Helm)
2. **Layer 2 ‚Äî SSE Event Channel** (how the runtime talks to the UI in real time)
3. **Layer 3 ‚Äî UI Directives** (how Helm manipulates the UI)

Plus two cross-cutting concerns:

- **Smart Routing** (per ADR-011 ‚Äî how the UI picks which runtime to talk to)
- **Auth** (per T0.A8 ‚Äî how requests are authenticated)

## Layer 1 ‚Äî Request/Response Contract

### Endpoints

| Endpoint | Method | Auth | Purpose |
|---|---|---|---|
| `/invoke/helm_prime` | POST | Bearer token required | Send a message to Helm, receive a structured response |
| `/health` | GET | No auth (exempt) | Runtime health status ‚Äî UI polls for status indicator, Docker/Render healthcheck |
| `/config/agents` | GET | Bearer token required | Agent roster ‚Äî UI reads for Agent Status widget |
| `/events` | GET | Bearer token required | SSE event stream (see Layer 2) |

### POST `/invoke/helm_prime`

**Request:**

```json
{
  "user_message": "What's the status on Lane C?",
  "session_id": "8f3a2b1c-4d5e-6f7a-8b9c-0d1e2f3a4b5c",
  "turn_number": 3,
  "context": {
    "surface": "desktop_ui"
  },
  "project": "hammerfall-solutions"
}
```

| Field | Type | Required | Notes |
|---|---|---|---|
| `user_message` | string | Yes | The user's message text |
| `session_id` | UUID string | Yes | Generated by UI on first load, persisted in localStorage. One per device at T1. |
| `turn_number` | integer | Yes | Incremented per turn within a session. Starts at 0. |
| `context` | object | No | Optional metadata. `surface` indicates where the user is engaging. |
| `project` | string | No | Defaults to `"hammerfall-solutions"` if absent. Used for Supabase partition scoping. |

**No `agent` field in the request body.** The URL path `/invoke/helm_prime` is the agent selector. The body describes the message, not the destination.

**Response:**

```json
{
  "text": "Phase 1 is complete ‚Äî runtime boots clean after three infrastructure fixes. Phase 2 opens with the Prime handler build. I'm ready when you are.",
  "subsystems_invoked": ["projectionist", "archivist"],
  "ui_directives": [
    {
      "action": "open_widget",
      "widget": "agent_status",
      "target": "dock"
    }
  ]
}
```

| Field | Type | Always present | Notes |
|---|---|---|---|
| `text` | string | Yes | Helm's natural language response. Displayed in Console Chat. |
| `subsystems_invoked` | string[] | Yes | Which cognitive subsystems fired during this turn. Set semantics (no duplicates). |
| `ui_directives` | object[] | Yes | UI manipulation directives. Empty array `[]` if no UI action warranted. |

**`subsystems_invoked` contract:**

- Canonical values: `"projectionist"`, `"archivist"`, `"contemplator"` ‚Äî lowercase, matching `/config/agents` keys.
- **Prime never appears in the array.** He is always the voice. Empty array = pure Prime turn (no subsystems fired).
- Set semantics: no duplicates. If Archivist fires twice in one turn, it appears once.
- The UI renders these as small pills beneath the response (e.g., `PROJ` `ARCH`). Empty array = no pills.

**Fallback behavior:**

If the response is plain text (JSON parse fails), the UI treats the entire response as:

```json
{
  "text": "<the raw response>",
  "subsystems_invoked": [],
  "ui_directives": []
}
```

If the response is JSON but missing `text`, the UI logs a warning and displays the raw JSON as text. This ensures the Chat tab never breaks regardless of what the runtime returns.

### GET `/health`

**Response:**

```json
{
  "status": "healthy",
  "agents": {
    "helm_prime": "ok",
    "projectionist": "ok",
    "archivist": "ok",
    "contemplator": "ok"
  },
  "supabase": "connected",
  "outbox": {
    "queued_count": 0,
    "dead_letter_count": 0
  }
}
```

| Field | Notes |
|---|---|
| `status` | `"healthy"`, `"degraded"`, or `"unhealthy"` |
| `agents` | Per-agent status. `"ok"` or `"unreachable"`. |
| `supabase` | `"connected"` or `"disconnected"` |
| `outbox` | Outbox queue state. Non-zero `dead_letter_count` ‚Üí `status` degrades. |

**This endpoint must stay boring.** No env vars, no secrets, no config details. Status booleans and counts only. `/health` is exempt from auth (T0.A8) and publicly accessible for Docker/Render healthchecks.

**UI polling:** Every 5 seconds. Three consecutive failures ‚Üí UI shows "Helm is unreachable." Configurable via a UI constant.

### GET `/config/agents`

**Response:**

```json
{
  "agents": {
    "helm_prime": {
      "provider": "anthropic",
      "model": "claude-opus-4-6"
    },
    "projectionist": {
      "provider": "ollama",
      "model": "qwen3:4b",
      "base_url": "http://ollama:11434"
    },
    "archivist": {
      "provider": "ollama",
      "model": "qwen3:14b",
      "base_url": "http://ollama:11434"
    },
    "contemplator": {
      "provider": "ollama",
      "model": "qwen3:14b",
      "base_url": "http://ollama:11434"
    }
  }
}
```

Used by the Agent Status widget to display agent names, providers, and models. The UI does NOT use this to make routing decisions ‚Äî routing is the runtime's job.

### Session Management

- UI generates a UUID v4 on first load: `crypto.randomUUID()`
- Persisted in `localStorage` under key `helm_session_id`
- Sent with every `/invoke` request
- New session = new UUID (user clicks "New Session" or clears storage)
- **Per-device sessions at T1.** Two devices = two sessions, two conversation threads. This is a known T1 limitation. Cross-device session identity moves to Supabase in Stage 2 when user auth exists.

### Error Handling

| Scenario | UI behavior |
|---|---|
| `/health` fails 3 consecutive times | Show "Helm is unreachable" in Console status bar |
| `/invoke` returns 401 | Show "Authentication failed ‚Äî check your API token" |
| `/invoke` returns 429 | Show "Rate limited ‚Äî retry in {Retry-After}s" |
| `/invoke` returns 500 | Show "Helm encountered an error" with retry button |
| `/invoke` times out (>30s) | Show "Helm is taking longer than expected" with cancel/retry |
| Network error (no response) | Queue message locally, retry on next successful `/health` |

Messages queued during an outage are sent in order when the runtime returns. The UI maintains a local send queue.

## Layer 2 ‚Äî SSE Event Channel

### Endpoint

`GET /events` ‚Äî Server-Sent Events stream. The UI subscribes when the Console opens and unsubscribes when the Console closes.

**Auth:** Bearer token required. Standard `EventSource` does not support headers. Use `fetch-event-source` library (or equivalent polyfill) to add the `Authorization` header.

```javascript
import { fetchEventSource } from '@microsoft/fetch-event-source';

fetchEventSource(`${runtimeUrl}/events`, {
  headers: {
    'Authorization': `Bearer ${apiToken}`,
    'x-correlation-id': sessionCorrelationId
  },
  onmessage(event) {
    const data = JSON.parse(event.data);
    routeEvent(data);
  },
  onerror(err) {
    setConnectionStatus('disconnected');
  }
});
```

### Event Schema

Every SSE event has this shape:

```json
{
  "id": "evt_00042",
  "type": "agent_completed",
  "agent": "projectionist",
  "action": "frame_write",
  "timestamp": "2026-04-21T22:14:03.482Z",
  "correlation_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "payload": {
    "latency_ms": 210,
    "frame_id": "8f3a2b1c-..."
  }
}
```

| Field | Type | Notes |
|---|---|---|
| `id` | string | Incrementing event ID. Used for `Last-Event-ID` reconnection (T4.3). |
| `type` | string | Event type from the vocabulary below. |
| `agent` | string | Which agent/component produced this event. |
| `action` | string | Human-readable action description. |
| `timestamp` | ISO 8601 string | UTC. |
| `correlation_id` | string | Traces back to the originating request. Matches `x-correlation-id` header. |
| `payload` | object | Type-specific data. May be empty `{}`. |

### Event Types

| Event type | Agent | When it fires | Payload |
|---|---|---|---|
| `agent_invoked` | Any | Handler called | `{}` |
| `agent_completed` | Any | Handler returned | `{latency_ms: number}` |
| `agent_error` | Any | Handler threw exception | `{error: string, error_type: string}` |
| `contemplator_pass_started` | Contemplator | Deep pass (think mode) begins | `{}` |
| `contemplator_pass_completed` | Contemplator | Deep pass ends | `{summary: string, belief_patches: number, curiosity_flags: number}` |
| `frame_written` | Projectionist | New frame committed to Supabase | `{frame_id: string}` |
| `frame_migrated` | Archivist | Cold frame drain completed | `{count: number}` |
| `belief_updated` | Contemplator | Belief strength changed | `{belief_id: string, slug: string, delta: number}` |
| `curiosity_flag` | Contemplator | New curiosity generated | `{curiosity_id: string, text: string}` |
| `personality_read` | Prime | Personality scores loaded at session start | `{}` |
| `system_health` | Runtime | Health check or system state change | `{status: string, details: object}` |
| `session_started` | Runtime | New session opened | `{session_id: string}` |
| `archivist_drain_failed` | Runtime | Async drain error | `{error: string, severity: "error"}` |
| `client_lagging` | Runtime | SSE queue overflow for this client | `{missed_count: number}` |
| `circuit_breaker_state` | Runtime | Supabase circuit breaker changed | `{state: "open" | "closed", component: "supabase"}` |
| `signal_dual_write_failed` | Runtime | Signal mirror write failed | `{error: string, entry_id: string}` |
| `outbox_dead_letter` | Runtime | Entry moved to dead letter after max retries | `{entry_id: string, table: string}` |

### SSE Backpressure

Server maintains `asyncio.Queue(maxsize=256)` per connected client. On overflow: **drop-oldest** (not drop-newest). When overflow occurs, the server emits a `client_lagging` event to that specific client with `missed_count`.

**Why drop-oldest:** Drop-newest creates "stuck spinner" bugs ‚Äî `agent_invoked` fires but `agent_completed` gets dropped, so the UI shows Helm as permanently processing. Drop-oldest keeps the UI showing current state.

### SSE Reconnection

If the SSE connection drops, the client reconnects with `Last-Event-ID` header set to the last received event `id`. The server replays missed events from a ring buffer (last 100 events per client session).

If the buffer does not contain the requested `Last-Event-ID` (overflowed), the server sends a `replay_unavailable` event. The UI knows to refresh widget state from Supabase rather than relying on the event stream for catch-up.

Full specification in T4.3.

### Node State Mapping

SSE events drive the Helm node's visual state in the canvas:

| SSE Event | Node State | Visual |
|---|---|---|
| `agent_invoked` where agent = `helm_prime` | Processing | Blue pulse animation |
| `contemplator_pass_started` | Contemplating | Amber glow animation |
| `agent_completed` where agent = `helm_prime` | Idle | Return to default particle state |
| `agent_error` | Error | Red flash, 3-second decay back to idle |

Other events (frame_written, belief_updated, etc.) do not change node state ‚Äî they are informational, routed to Activity/System tabs only.

### Connection Status Indicator

The Console header displays a connection badge:

| State | Badge | Color | When |
|---|---|---|---|
| Connected | `LIVE` | Green | SSE stream is active and receiving events |
| Disconnected | `DISCONNECTED` | Red | SSE stream dropped, reconnection in progress |
| Mocked | `MOCKED` | Amber | Running on mock data (no SSE connection attempted) |
| Lagging | `LIVE (behind)` | Amber | `client_lagging` event received ‚Äî UI missed some events |

## Layer 3 ‚Äî UI Directives (Helm's Hands)

### What This Is

Helm can manipulate the UI by including directives in his response. "Show me the agent status" ‚Üí Helm opens the Agent Status widget. "Hide the personality panel" ‚Üí Helm closes it. This is Helm operating prebaked, known components ‚Äî not composing novel interfaces (that is Stage 4 polymodal).

### What This Is NOT

**Polymodal.** Polymodal (Stage 4) is a library of primitives (lists, graphs, charts, numbers, images) that Helm assembles into previously undefined widgets on the fly. Layer 3 is the prerequisite ‚Äî you cannot compose novel interfaces if you cannot even open existing ones. But Layer 3 itself is a fixed vocabulary operating on known components.

### Directive Schema

Each directive in the `ui_directives` array:

```json
{
  "action": "open_widget",
  "widget": "agent_status",
  "target": "dock"
}
```

### Directive Vocabulary

| Action | Parameters | What it does |
|---|---|---|
| `open_widget` | `widget` (string), `target` ("dock" or "canvas") | Opens a prebaked widget in the specified location |
| `close_widget` | `widget` (string) | Closes the widget wherever it is currently open |
| `minimize_widget` | `widget` (string) | Minimizes widget to pill in Console header |
| `expand_widget` | `widget` (string) | Expands a docked widget to fill the right pane |
| `highlight_entry` | `widget` (string), `entry_id` (string) | Opens the widget (if not open), scrolls to the specified entry, and highlights it |
| `open_split` | `tab` ("activity" or "system") | Opens Console split view with the specified tab alongside chat |
| `close_split` | ‚Äî | Closes split view, restores widget dock |

### Widget Identifiers

| Identifier | Widget |
|---|---|
| `agent_status` | Agent Status ‚Äî subsystem health, latency, last invocation |
| `core_beliefs` | Core Beliefs ‚Äî belief list with strengths and observation history |
| `personality` | Personality ‚Äî six dimension sliders with translations |
| `memory` | Memory ‚Äî memory entries with type, content, confidence |
| `entities` | Entities ‚Äî entity list with types, aliases, relationship counts |
| `signals` | Signals ‚Äî pattern aggregations with observation counts |
| `logs` | Logs ‚Äî session/contemplator/system tabs |

### When Helm Emits Directives

The directive decision lives in Prime's reasoning ‚Äî not a classifier, not a router. Prime reads the user's message, decides if a UI action would genuinely help the conversation, and includes directives when warranted.

**Examples:**

| User says | Helm does |
|---|---|
| "Show me the beliefs" | Responds in text AND emits `open_widget` + `core_beliefs` + `dock` |
| "What's the agent status?" | Answers the question in text AND emits `open_widget` + `agent_status` |
| "How direct am I set?" | Answers the specific score in text. May or may not open personality panel (judgment call). |
| "Tell me about the roadmap" | Pure text response. No directives. The roadmap is not a widget. |
| "Hide all the panels" | Brief acknowledgment AND emits `close_widget` for each currently open widget |
| "Open the activity stream" | Confirms AND emits `open_split` + `activity` |

**Most responses have empty `ui_directives`.** Directives are the exception, not the rule. Helm should not open widgets unprompted unless the conversation strongly warrants it.

### Unknown Directives

If the UI receives a directive with an unrecognized `action`, it logs a warning and ignores it:

```javascript
default:
  console.warn(`[helm-ui] Unknown directive action: ${action}`);
```

This is forward-compatible ‚Äî new directive actions can be added to the runtime without breaking older UI versions.

## Smart Routing (per ADR-011)

### Overview

The UI supports two configured runtime URLs and picks the active one via a probe-and-failover state machine:

| Env var | Purpose | Default |
|---|---|---|
| `VITE_HELM_LOCAL_URL` | Laptop runtime | `http://localhost:8000` |
| `VITE_HELM_REMOTE_URL` | Cloud runtime (Render) | Empty ‚Äî set when T4.11 ships |

### State Machine

**State 1 ‚Äî Startup probe:**

1. UI hits `${LOCAL_URL}/health` with 2-second timeout
2. If 200 OK ‚Üí use local
3. If fail AND `REMOTE_URL` configured ‚Üí probe remote with 5-second timeout
4. If remote 200 OK ‚Üí use remote, show amber banner: "Using cloud Helm ‚Äî laptop runtime not detected"
5. If both fail ‚Üí show red error screen: "No Helm reachable"

**State 2 ‚Äî Active session, request fails:**

1. If currently on local AND `REMOTE_URL` configured ‚Üí auto-failover to remote
2. Show amber banner: "Couldn't reach laptop Helm ‚Äî switched to cloud Helm"
3. If currently on remote ‚Üí show red banner with retry button: "Cloud Helm unreachable"
4. If `REMOTE_URL` not configured ‚Üí show red banner: "Laptop Helm unreachable and no fallback configured"

**State 3 ‚Äî After failover, re-probe local:**

1. For the next 3 turns, re-probe `LOCAL_URL` alongside the active request
2. If local returns within 3 attempts ‚Üí silent switch back, amber banner: "Back on laptop Helm"
3. If still unreachable after 3 attempts ‚Üí prominent amber banner: "Still on cloud Helm ‚Äî laptop hasn't come back"
4. Stop auto-re-probing. User can click manual retry button in the banner.

**State 4 ‚Äî Per-session stickiness:**

Once a runtime is chosen (and any failovers resolved), the UI sticks with it for the SSE session. New chat session = new startup probe.

### Banner UX

- Banners are **non-modal** ‚Äî they do not interrupt the conversation
- Amber: informational ("switched to cloud", "back on laptop")
- Red: action needed ("no Helm reachable", "cloud unreachable after retry")
- All banners include a manual retry button that immediately re-probes local
- Banners dismiss automatically when the state resolves (e.g., "back on laptop" dismisses after 5 seconds)

### Failover During Active SSE Stream

If failover occurs while an SSE stream is active, the stream is aborted. The UI:

1. Closes the current `EventSource`
2. Opens a new `EventSource` to the new runtime URL
3. Shows banner noting the switch
4. Does NOT attempt to replay missed events across runtimes (the two runtimes have independent event histories)

Document in T1.7: failover during active stream aborts the stream + shows banner. User retries; new request goes to fallback runtime. No silent partial responses.

## Auth Contract (per T0.A8)

### Bearer Token

All protected endpoints require an `Authorization: Bearer <token>` header.

```
Authorization: Bearer <HELM_API_TOKEN value>
```

- Token source: `VITE_HELM_API_TOKEN` in `helm-ui/.env`
- This is a **coarse boundary token**, not a user-auth token. It excludes random pings, not sophisticated attackers.
- Same token shared across all environments at T1. Per-deploy rotation is Stage 2.
- `/health` is exempt ‚Äî no auth required.

### Token Failure Responses

| Status | Meaning | UI behavior |
|---|---|---|
| 401 | Missing or invalid token | Show "Authentication failed" error. Do not retry automatically. |
| 500 | `HELM_API_TOKEN` not configured on server | Show "Helm misconfigured" error. This is an operator issue, not a user issue. |

### Correlation ID

Every request from the UI includes an `x-correlation-id` header:

```
x-correlation-id: <session-scoped UUID>
```

- Generated once per session (same lifetime as `session_id`)
- The server echoes it back in the response header
- The server propagates it to all structured logs and SSE events during that request
- The UI can display it in a debug panel for support/troubleshooting

## Health Poll Cadence

| Parameter | Value | Notes |
|---|---|---|
| Poll interval | 5 seconds | Configurable via UI constant |
| Failure threshold | 3 consecutive failures | After 3 fails ‚Üí "unreachable" |
| Recovery | First successful poll ‚Üí clear "unreachable" | Immediate recovery, no hysteresis |
| Timeout per poll | 5 seconds | Shorter than the poll interval |

## Rate Limiting (T4.2)

The `/invoke` endpoint has a rate limit (default 60 requests/minute). When exceeded:

```
HTTP 429 Too Many Requests
Retry-After: 12
```

The UI reads the `Retry-After` header and shows: "Rate limited ‚Äî retry in 12s." A countdown timer is displayed. The send button is disabled until the timer expires.

## What This Spec Does NOT Cover

- **Supabase query patterns** ‚Äî how widgets query tables directly. See `docs/stage1/schema-reference.md` (T2.8).
- **Polymodal interface** ‚Äî Helm composing novel widgets from primitives. Stage 4.
- **Multi-user auth** ‚Äî OAuth, SSO, per-user identity. Stage 2.
- **Cross-device session sync** ‚Äî sharing session state across devices. Stage 2 (requires user auth).
- **Streaming responses** ‚Äî `/invoke` currently returns a complete response. Streaming (token-by-token) is a Stage 2 optimization.
- **WebSocket alternative to SSE** ‚Äî SSE is sufficient for T1's unidirectional event stream. WebSocket considered for Stage 2 if bidirectional communication is needed.
