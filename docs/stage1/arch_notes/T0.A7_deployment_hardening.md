# T0.A7 — Deployment Hardening

**Tier:** ARCH
**Phase:** 0A — Repo & Operational Foundation
**Spec section:** [Helm_T1_Launch_Spec_V2.md §T0.A7](../Helm_T1_Launch_Spec_V2.md)
**Status:** Pre-build design note for architect review.

---

## What I'm building

A multi-stage `services/helm-runtime/Dockerfile` plus `docker-compose.yml` updates that make the runtime container production-shaped (non-root user, read-only root filesystem, capability drops, healthcheck, resource limits, hash-pinned deps). Not yet production-deployed — that's T4.4's call (ADR-003) and T4.11's build.

## Why this shape

Three forces converged on this design:

1. **The container will run on Render free tier eventually (T4.11).** Render kills containers that misbehave. A container that runs as root, has writable root, and holds full Linux capabilities is a footgun on shared infra. Build it right at T0.A7 so we don't refactor it under deadline pressure later.
2. **Hash-pinned deps is the only durable way to keep CI reproducible.** Floating versions break "rebuild from main" six months in. `pip-compile` with hashes is the cheapest discipline that still works.
3. **Healthcheck is non-negotiable for Render's deploy promotion.** Render uses container healthchecks to decide if a deploy is good. Defining it at T0.A7 means the container is Render-ready before T4.11 even starts.

## Alternatives considered + rejected

- **Single-stage Dockerfile (simpler).** Rejected — image is ~3x larger, leaks pip/build artifacts into the runtime, and slows every cold start on Render.
- **Run as root with `--user 1000:1000` at compose time.** Rejected — the user has to exist in the image for filesystem permissions to make sense; declaring it in the Dockerfile is the only correct path.
- **Skip read-only root + cap_drop until T4.11.** Rejected — these are 5 lines of YAML and they catch real misconfigurations during dev (e.g., something tries to write to `/etc`). Defer-cost > do-it-now-cost.
- **Distroless base instead of `python:slim-bookworm`.** Considered — distroless is smaller and more secure, but it makes shelling in for debug nearly impossible. For a single-dev project, debuggability wins. Re-evaluate at Stage 2 if runtime is stable.

## Interfaces / contracts

- **T0.A12 (CI: Container Build + GHCR Publish)** depends on this Dockerfile being buildable in CI.
- **T0.A11 (Runtime Guardrails)** depends on resource limits being set so the rate-alarm/dollar-cap logic has a sane container budget.
- **T4.4 (Dev Deployment Decision)** consumes the production-shaped image to decide where to deploy.
- **T4.11 (Persistent Dev Deployment)** consumes the same image; healthcheck must match Render's expectations.
- **The `pip-compile` workflow becomes mandatory** for every dep change after T0.A7 lands. T0.A14 (Dependency Automation) wires this into Renovate/Dependabot.

## Risks + open questions for architect

- **Read-only root + tmpfs `/tmp` — does anything we depend on write outside `/tmp`?** I think no, but Uvicorn workers, Python's logging module, and any future cache layer all need to be checked. Architect: do you know of anything in the SDK or FastAPI stack that writes to a non-`/tmp` path by default?
- **`mem_limit` and `cpus` values not yet specified in spec.** I plan to start with `mem_limit: 512m`, `cpus: 1.0` for local dev (matches Render free tier ceiling roughly). Architect: do you want different starting values, or wait for T4.8 bench data to set them?
- **Healthcheck endpoint is `/health` returning 200.** T0.A6 (Observability) defines what `/health` actually checks (Supabase reachable? all 4 agents loaded?). If `/health` becomes expensive, the 30s healthcheck interval will rate-limit the runtime against itself. Worth a flag.
- **Image base pin is `python:3.12.4-slim-bookworm`.** Architect: any preference for a specific patch line we should track, or is "latest 3.12.x slim" acceptable?

## What "done" looks like at the gate

- `docker build` produces an image that runs as `helm` (non-root), starts uvicorn, and `/health` returns 200.
- `docker compose up` brings the stack up with read-only root and cap_drop — and the runtime stays up across a restart of the container.
- `pip-compile` produces a `requirements.txt` with hashes; CI fails if `requirements.txt` is out of sync with `requirements.in`.
- The Dockerfile has zero secrets baked in; all env via `.env` mount.
- A SITREP at PR open notes any deviations from this design note.
