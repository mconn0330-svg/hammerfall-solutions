# T0.A8 — API Auth

**Tier:** ARCH
**Phase:** 0A — Repo & Operational Foundation
**Spec section:** [Helm_T1_Launch_Spec_V2.md §T0.A8](../Helm_T1_Launch_Spec_V2.md)
**Status:** Pre-build design note for architect review.

---

## What I'm building

A static bearer-token gate on `/invoke/*`, `/events`, and `/config/*` endpoints. Token in `HELM_API_TOKEN` env var; FastAPI `Depends(require_token)` on each protected route; `/health` exempt for Docker/Render healthchecks. Frontend integration deferred to T3.4.

## Why this shape

Three forces:

1. **Even on localhost, anonymous endpoints are wrong.** The runtime takes a model-invocation request and spends Pro Max quota / external API budget on it. A drive-by curl from any process on the box shouldn't be able to drain that.
2. **Static bearer is the lowest-cost auth that's actually a boundary.** It excludes random pings without requiring an identity provider, JWT validation, or session state. The threat model at T1 is "noise on the box," not "sophisticated attacker" — match the bar to the threat.
3. **`Depends()` is the FastAPI-idiomatic enforcement point.** Decorating each route, not middleware. Middleware is harder to reason about per-route exemptions (we need `/health` open) and harder to test in isolation.

## Alternatives considered + rejected

- **No auth until T2/Stage 2 user auth.** Rejected — the runtime starts spending Pro Max quota the day it boots; "I'll add auth later" is how budget incidents happen.
- **Middleware-based gate with allowlist.** Rejected — two configuration surfaces (allowlist + decorators) is one too many; route-level `Depends` is single-source.
- **mTLS / cert-based auth.** Rejected — wildly disproportionate to the T1 threat model; cert lifecycle would be a second runbook on day one.
- **Auth in NGINX/reverse-proxy layer.** Rejected — we don't have a reverse proxy at T1; adding one to host the auth would be more architecture, not less.
- **Rate-limit-only (no auth).** Rejected — rate limiting is T4.2 and is additive to auth, not a replacement; "slow noise" is still budget drain.

## Interfaces / contracts

- **T0.A11 (Runtime Guardrails)** assumes `/invoke` requests are attributable to a known caller; auth establishes that minimal attribution.
- **T0.A12 (CI: Container Build)** must inject `HELM_API_TOKEN` into the test container so smoke tests can hit `/invoke` in CI.
- **T3.4 (Frontend integration)** consumes the same `HELM_API_TOKEN` via `VITE_HELM_API_TOKEN` and adds the `Authorization: Bearer ...` header in the fetch wrapper.
- **T4.2 (Rate Limiting on /invoke)** layers on top of auth; rate keys can use the bearer token as the identity.
- **`docs/runbooks/0001-api-token-rotation.md`** is the operational interface — must exist before T0.A8 PR closes.

## Risks + open questions for architect

- **`/health` exemption is the only opening.** If `/health` ever returns sensitive data (e.g., env-derived config), the exemption becomes a leak. T0.A6 (Observability) defines `/health` payload — keep it boring (status + agent count + Supabase reachable, no env data). Worth re-validating at T0.A6 review.
- **Token in `helm-ui/.env` is client-visible.** Spec already calls this out: `VITE_HELM_API_TOKEN` is a coarse boundary, not a secret. Architect: do you want a rotated-per-deploy approach now (T4.11 issues a fresh token on each deploy and bakes into the UI build), or accept "one shared token until Stage 2 user auth"?
- **No token-rotation tooling at T1.** Manual `openssl rand -hex 32`, edit `.env`, restart. Acceptable for T1; flag for T4.x runbook polish.
- **`HTTP_500` when token unconfigured.** If env is missing, the route 500s instead of 401s. This is correct (misconfig vs. unauthorized), but it does mean a bad deploy will surface as 500s on every protected route. Worth a smoke test in T0.A12 CI pipeline.

## What "done" looks like at the gate

- Curl without bearer to `/invoke/helm_prime` returns 401.
- Curl with wrong bearer returns 401.
- Curl with correct bearer returns expected response.
- Curl to `/health` with no auth returns 200.
- Runbook `0001-api-token-rotation.md` exists with rotation steps.
- CI smoke test exercises both 401 and 200 paths.
- A SITREP at PR open notes any deviations from this design note.
