# T0.B1 — Memory Module Core

**Tier:** ARCH
**Phase:** 0B — Memory Foundation
**Spec section:** [Helm_T1_Launch_Spec_V2.md §T0.B1](../Helm_T1_Launch_Spec_V2.md)
**Status:** Pre-build design note for architect review.

---

## What I'm building

A `services/helm-runtime/memory/` Python module that becomes the single in-process write/read API for Helm's brain. v1's external `brain.sh` shell helper is removed (T0.B6). Module owns: `MemoryType` enum, Pydantic settings (env-loaded), retry+backoff client, circuit breaker with observability hooks, write/read helpers, OpenTelemetry spans on every write. Built so Tier 2 (T0.B7) and Tier 3 (post-T1) can extend it without surgery.

## Why this shape

1. **The shell-helper-as-IPC pattern was the single biggest architectural gap in v1.** Subprocess spawn per memory write meant: no transactions, no retries with shared state, no atomic dual-write, no observability spans. An in-process module collapses 5 problems into one library.
2. **The abstraction is exercised by T0.B7, validated by Tier 3 deferral.** This is the spec's deliberate test: if T0.B7 has to do >150 lines per type, the abstraction failed and we revisit before continuing. That's a hard contract, not a hope.
3. **Pydantic settings with `env_prefix` is the only durable env-loading approach.** Bare `os.environ` reads scattered through code rot quickly; Pydantic gives one place to see every config knob and enforce types.
4. **Circuit breaker + observability hooks together, not separately.** A breaker that opens silently is worse than no breaker. The `_on_state_change` callback is what makes the System tab light up when the brain is unreachable — turning a silent failure into a visible one.
5. **OS-aware default for `outbox_path` (`Path.home() / ".helm" / "outbox.db"`).** Maxwell's dev env is Windows; v1's `/tmp/helm-outbox.jsonl` would have broken on day one of T1 execution.

## Alternatives considered + rejected

- **Keep `brain.sh`, wrap it in Python.** Rejected — the failure modes (subprocess timeouts, stdout parsing, no shared state) are exactly what we're trying to leave behind.
- **Single big `memory.py` module.** Rejected — testing surface is unbearable; splitting into `_models.py`, `_settings.py`, `_client.py`, `_writer.py`, `_circuit_breaker.py` keeps each test file focused.
- **Type-specific write methods only (`write_frame`, `write_belief_update`, etc.) — no generic `memory.write(...)`.** Rejected — T0.B7 would have to add three new methods + change every caller; the generic core with type-specific thin wrappers keeps both shapes available.
- **Use `tenacity` library for retry instead of hand-rolling.** Considered. Tenacity is mature but pulls a dep for ~40 lines of logic. Defer the dep; if retry config grows, swap in tenacity at Stage 2.
- **`asyncio` from day one.** Considered — FastAPI is async, but the memory writes are single-shot, not high-concurrency. Sync API is simpler and fits the call sites; async wrapper added later if needed.
- **Use `httpx` async client.** Going with `httpx.Client` (sync). Simplifies the circuit breaker state machine. Async added at T2 if request volume justifies it.

## Interfaces / contracts

- **T0.B2 (Outbox Pattern)** wraps `memory.write` with the dual-write to SQLite. T0.B1 must expose the write path as a function the outbox can wrap, not as a method buried in a class.
- **T0.B3 (Migrate In-Process Agents)** consumes `memory.write/read` directly — this is the consumer that proves the API shape works in real call sites.
- **T0.B6 (Prompt Rewrite)** rewrites every agent prompt's `bash scripts/brain.sh ...` reference into `memory.write(...)` Python pseudocode. The API shape T0.B1 ships is the surface those prompts will reference forever; getting the names right matters.
- **T0.B7 (Tier 2 Brain Types)** is the abstraction validation. Each new type gets ~50 lines: migration + enum line + thin helper + tests. >150 lines means revisit T0.B1.
- **T2.6, T2.7 (dual-write hooks)** consume the same `_on_state_change` callback wired into `CircuitBreaker` so SSE can surface state.
- **T0.A6 (Observability)** consumes the OpenTelemetry spans T0.B1 emits on every write.

## Risks + open questions for architect

- **`MemoryType` enum is the most-mutated surface.** Every new brain type appends a value. Convention: never rename, never remove, append only. If we need to deprecate, mark in comments and stop writing — never break existing readers. Architect: do you want this as an explicit ADR, or comment-in-code is enough?
- **Circuit breaker thresholds (5 failures, 30s cooldown) are guesses.** No real signal yet on what Supabase outage looks like. Worth keeping configurable from day one (already in settings) and re-tuning at T4.5 SITREP.
- **`memory.read` API not fully sketched in spec.** Spec covers writes; reads are needed for Routine 0 (last 30 behavioral, last 10 scratchpad, delta query). I'll design the read API in T0.B1 — likely `memory.read(project, agent, type=None, since=None, limit=None)` plus `memory.session_start_context()` as a higher-level convenience for Routine 0. Architect: any preference for a different shape?
- **Retry behavior on partial Supabase failure.** Supabase REST returning 5xx is retried; 4xx is not. But what about timeout-mid-write where the write may or may not have committed? Plan: timeout = treat as failure, queue to outbox (idempotent on `id` UUID generated client-side); duplicate writes deduplicated by `id`. Architect: agree with client-side UUID + dedup, or prefer server-side?
- **Test coverage target.** Spec doesn't specify a numeric target for memory module. I'll aim for >90% line coverage on the core (`_writer`, `_client`, `_circuit_breaker`) and >70% on the periphery. Architect: hard floor different?

## What "done" looks like at the gate

- `memory.write(project="hammerfall-solutions", agent="helm_prime", memory_type=MemoryType.BEHAVIORAL, content="...")` produces a row in Supabase `helm_memory`.
- `memory.read(...)` returns rows for the agreed read API.
- `memory.session_start_context()` returns the dict shape T0.B2 needs (queued_count, oldest_queued_at, dead_letter_count).
- A simulated Supabase outage opens the circuit breaker, emits an observability event, fails fast on subsequent writes — and recovers after cooldown.
- `tests/test_models.py`, `tests/test_settings.py`, `tests/test_client.py`, `tests/test_writer.py` all green; coverage meets target.
- Module imports cleanly with no env vars set (settings load lazily, not at import).
- A SITREP at PR open notes any deviations from this design note.

## Architect review — 2026-04-24

**Verdict:** APPROVE.

**Resolved open questions:**
- **`MemoryType` enum append-only convention:** Comment-in-code is sufficient. No separate ADR needed.
- **Circuit breaker thresholds (5 failures, 30s cooldown):** Keep configurable via settings. Re-tune at T4.5 SITREP from real signal.
- **`memory.read` API shape:** `memory.read(project, agent, type=None, since=None, limit=None)` plus `memory.session_start_context()` confirmed. The convenience method is important — agents should not assemble their own session-start context from raw reads.
- **Timeout-mid-write:** Client-side UUID + dedup confirmed. Standard upsert-with-client-IDs pattern for exactly-once delivery over unreliable networks.
- **Test coverage targets:** >90% on core (`_writer`, `_client`, `_circuit_breaker`), >70% on periphery — confirmed.
- **Sync vs async:** Start sync (`httpx.Client`); add async wrapper at Stage 2+ if request volume justifies.
