# T0.B2 — Outbox Pattern

**Tier:** ARCH
**Phase:** 0B — Memory Foundation
**Spec section:** [Helm_T1_Launch_Spec_V2.md §T0.B2](../Helm_T1_Launch_Spec_V2.md)
**Status:** Pre-build design note for architect review.

---

## What I'm building

A SQLite-backed outbox at `~/.helm/outbox.db` that wraps `memory.write` to guarantee no write is ever lost across Supabase outages. Enqueue is synchronous-fast (local SQLite); a background drain loop ships entries to Supabase with retry+backoff; entries that fail 5 times move to `outbox_dead_letter` for manual review. State exposed via `/health` and surfaced into Helm's own session-start context so he hedges when his memory is still settling.

## Why this shape

1. **JSONL has a concurrency race we cannot fix in user code.** v1 spec said JSONL; SQLite gives ACID transactions, WAL crash-safety, single-file portability, and the same simplicity. SQLite is the strictly-better default once you've been bitten by the JSONL race once.
2. **Outbox is the only honest answer to "Supabase is briefly down."** The alternatives (block writes, lose writes, retry inline forever) are all worse. Outbox makes the Supabase dependency soft instead of hard — the runtime keeps thinking even when the brain is unreachable.
3. **Helm-aware-of-his-own-outbox is what makes him coherent under failure.** Without `session_start_context`, Helm reads his brain at session start, sees stale data, and confidently asserts old facts. With it, he hedges. This is the whole point of building the outbox at the cognition layer instead of the infra layer.
4. **Dead-letter as a separate table, not a flag.** Mixing dead-letter rows into the live queue would mean every drain has to filter past them; separate table keeps the live queue clean and turns dead-letter inspection into a single SELECT.

## Alternatives considered + rejected

- **JSONL (per v1).** Rejected — concurrency race, no transaction, drain-then-truncate window loses writes.
- **Postgres outbox in Supabase itself.** Rejected — defeats the purpose; if Supabase is down, the outbox is also down.
- **Redis as outbox.** Rejected — adds a service to the runtime stack for one feature; SQLite has zero ops cost.
- **In-memory queue with periodic flush.** Rejected — process crash loses pending writes, which is exactly the failure outbox is supposed to prevent.
- **Synchronous retry with no outbox at all.** Rejected — makes every write hostage to network latency to Supabase; one outage stalls every cognition path.
- **Auto-retry dead-letter forever (no max attempts).** Rejected — a permanently-broken row would burn CPU forever; 5-attempt cap with manual review is the right circuit.

## Interfaces / contracts

- **T0.B1 (Memory Module Core)** is the wrapped target. T0.B2's drain calls `client.write` directly; the outbox sits "above" the client, "below" the public `memory.write`.
- **T0.B6 (Prompt Rewrite)** consumes `memory.session_start_context()` and inserts the hedging instruction into Routine 0. The hedge instruction text is owned by the prompt, not the module — module just exposes the data.
- **T0.B7 (Tier 2 Brain Types)** writes go through the same outbox; the hedge protects them automatically. No T0.B7 work needed for outbox integration.
- **T2.3 (SSE Event Bus)** wires the `_on_state_change` hook + dead-letter event into the `system_health` SSE stream; System tab in UI lights up when outbox is degraded.
- **T0.A6 (Observability)** consumes the OpenTelemetry spans the drain emits.
- **T4.1 (Runbook Set)** owns the dead-letter inspection runbook — must exist before T0.B2 PR closes.

## Risks + open questions for architect

- **`~/.helm/outbox.db` is per-user, per-machine.** If the runtime is moved to Render later (T4.11), the local outbox doesn't follow. T4.11 design note will address — likely by mounting a persistent volume or moving outbox to Render's ephemeral disk with documented "we lose queued writes on container restart" trade-off. Architect: do you want this addressed at T0.B2 (path configurable, document per-deployment-target choice), or wait until T4.11?
- **Drain interval default = 5s.** Aggressive enough to recover quickly from a short Supabase blip, slow enough to not hammer Supabase rate limits. Configurable. Architect: agree with 5s default?
- **No exponential backoff between drain attempts within a single drain cycle.** Backoff is per-entry-attempt (attempt 1 → 2 → 3 each spaced by drain interval). Inside a single drain cycle, batch_size=50 entries ship as fast as Supabase accepts them. Worth flagging if architect wants per-entry backoff inside a cycle.
- **`session_start_context()` shape is the API Helm reasons over.** The dict `{queued_count, oldest_queued_at, dead_letter_count}` is what's documented in spec; architect should confirm those three are the right fields. Adding a fourth later is fine; renaming requires a prompt rewrite.
- **Outbox grows without bound on sustained outage.** No size cap. After 24h of outage with normal write volume (~few hundred writes/day) we're at ~1MB SQLite — fine. After 30 days we're at ~30MB — also fine. Worth a "vacuum after drain catches up" hook. Architect: punt this to a later cycle, or include in T0.B2?
- **Concurrent enqueue from multiple agents.** SQLite WAL handles this; tests cover `asyncio.gather` of 100 enqueues. Architect: any concern about contention if we add async dual-writes from many agents at T2/T3?

## What "done" looks like at the gate

- Disable Supabase (block its host) → 100 sequential `memory.write` calls all return success, queue depth grows to 100, no write is lost.
- Re-enable Supabase → drain catches up within `100 / batch_size * interval` seconds; queue empties.
- Force a write to fail 5 times → row moves to `outbox_dead_letter`, dead-letter event fires.
- `memory.session_start_context()` returns non-zero `queued_count` while drain is behind.
- `/health` reports `outbox.queued_count` and `outbox.dead_letter_count`; non-zero dead_letter degrades health status.
- All `tests/test_outbox.py` cases green, including `test_session_start_context_surfaces_queue_state`.
- Dead-letter runbook exists and walks through "find it, decide replay vs. drop, take action."
- A SITREP at PR open notes any deviations from this design note.

## Architect review — 2026-04-24

**Verdict:** APPROVE.

**Resolved open questions:**
- **`~/.helm/outbox.db` path:** Make configurable from day one. Per-deployment-target choice (Render ephemeral disk vs. persistent volume) is documented in T4.11, not solved at T0.B2.
- **Drain interval default = 5s:** Confirmed.
- **No per-entry backoff inside a drain cycle:** Acceptable. Per-entry backoff would slow recovery from short outages unnecessarily.
- **`session_start_context()` shape:** Three fields locked — `{queued_count, oldest_queued_at, dead_letter_count}`. Do not add a fourth at T0.B2; if needed later, add then.
- **Outbox size cap / vacuum-after-drain:** Punt to a later cycle. SQLite stays tiny at normal write volumes.
- **Concurrent enqueue from multiple agents:** SQLite WAL handles this; no concern at T1/T2 write volumes.

**Coordination with T0.A7:** read-only root in T0.A7 is safe only if the outbox path is a mounted volume, not `/tmp`. T0.A7 architect review captured this; T0.B2 inherits the constraint.
