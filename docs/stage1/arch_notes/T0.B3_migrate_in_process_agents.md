# T0.B3 — Migrate In-Process Agents

**Tier:** ARCH
**Phase:** 0B — Memory Foundation
**Spec section:** [Helm_T1_Launch_Spec_V2.md §T0.B3](../Helm_T1_Launch_Spec_V2.md)
**Status:** Pre-build design note for architect review.

---

## What I'm building

Replace every direct `supabase_client.insert/post` call inside agent code with `memory.write(...)`. All four agents (helm_prime, projectionist, archivist, contemplator) become consumers of the T0.B1 module. Reads are migrated progressively per a documented disposition table — writes unify now; some reads stay in `read_client` (renamed at T0.B6) until Stage 2.

## Why this shape

1. **Writes-first is the right risk profile.** Writes touch transaction safety, outbox, observability — areas where divergence is dangerous. Reads are idempotent and the divergence cost is mostly cosmetic. Migrate the dangerous half now; let the safe half move when convenient.
2. **A documented disposition table prevents drift.** Without an explicit row-by-row decision (Frame read → moves; personality read → stays), the migration becomes "what feels right today." Six months in, nobody remembers why some reads moved and some didn't. The table is the institutional memory.
3. **Renaming `supabase_client.py` → `read_client.py` (at T0.B6) is the cheapest disambiguation.** No behavior change; just makes the file's purpose unmistakable. Future contributors don't need to read the disposition table to know that file is read-only.
4. **Per-agent smoke test that mocks memory module proves the contract.** This is the test that catches "agent silently regressed to old code path." Each agent test asserts "this agent calls `memory.write` with these args" — pure interface check.

## Alternatives considered + rejected

- **Migrate writes AND reads in one shot.** Rejected — doubles the diff surface, doubles the verification burden, and the risk profile of reads doesn't justify the urgency.
- **Migrate per-agent in separate PRs.** Considered — would let each agent ship in isolation, but doubles the spec prep / SITREP overhead without reducing risk (the test surface is the same). One PR is correct.
- **Defer T0.B3 to after T0.B6 (prompt rewrite first).** Rejected — T0.B6 cites the new APIs in prompts; T0.B3 has to land first so the API surface is real before prompts reference it.
- **Keep both write paths live for a deprecation window.** Rejected — divergence is the bug we're fixing; a deprecation window guarantees divergence persists. Hard cutover.

## Interfaces / contracts

- **Depends on T0.B1 (Memory Module Core)** being complete and stable. T0.B3 is the first real consumer of the API; if T0.B1's API shape is wrong, T0.B3 surfaces it immediately.
- **Provides substrate for T0.B6 (Prompt Rewrite)** — the prompts reference the API surface this PR establishes.
- **Provides substrate for T0.B7 (Tier 2 Brain Types)** — when T0.B7 adds new brain types, agent prompts/code already write through `memory.write`, so T0.B7 helpers slot in cleanly.
- **Frame read by Archivist moves to `memory.read_frames()`** in T0.B3 itself (per disposition table). This is the only read migration in T0.B3; others wait.
- **`grep -r "supabase_client.insert\|supabase_client.post" agents/` returns zero** is the post-condition.
- **Tests at `tests/agents/test_<agent>.py`** become the long-lived contract that agents-call-memory-correctly.

## Risks + open questions for architect

- **Hidden write paths.** Some agents may write through paths grep doesn't catch (e.g., a method on a wrapping class that internally calls `supabase_client.post`). Plan: in addition to grep, run the runtime end-to-end and assert structured logs show every expected write coming through `memory.write` spans. Architect: any other write path you know of that isn't grep-able?
- **Frame read migration is the only read change in this PR.** Why this one? Because Archivist's frame-read is part of a tight write-then-read loop with `memory.write_frame`; co-locating both sides through the module avoids a write-here-read-there confusion. Other reads are split-but-stable. Architect: confirm this single carve-out is acceptable, or push for "all writes, zero reads" purity?
- **No backwards-compat layer.** If a third-party tool (none currently) was calling `supabase_client.insert`, it would break. We have no such consumers. Architect: fair to assume?
- **Test mock fidelity.** The smoke tests mock `memory.write` and check args. If `memory.write`'s signature evolves (T0.B7 adds optional kwargs), tests pass while agents may be missing the new kwargs. Mitigation: agent-level integration test (T2.9 simulation harness) catches this. Architect: acceptable to lean on T2.9 for that, or add tighter check at T0.B3?
- **Per-agent write coverage matrix not yet drafted.** I'll produce one as part of the PR — table of (agent, memory_type, frequency) — to verify nothing is dropped during migration. Architect: useful, or unnecessary?

## What "done" looks like at the gate

- `grep -r "supabase_client.insert\|supabase_client.post" agents/` returns zero hits.
- Live runtime invocation: each agent (Prime, Projectionist, Archivist, Contemplator) emits at least one `memory.write` span in structured logs during a hello-world session.
- Archivist's frame read goes through `memory.read_frames()`; old read path removed.
- Disposition table for reads lives in PR description (and gets archived in `docs/stage1/SITREPs/`).
- `tests/agents/test_helm_prime.py`, `test_projectionist.py`, `test_archivist.py`, `test_contemplator.py` all green; each asserts the expected `memory.write` call.
- A SITREP at PR open notes any deviations from this design note.
