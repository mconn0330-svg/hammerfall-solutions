# T0.A12 — CI: Container Build + GHCR Publish

**Tier:** ARCH
**Phase:** 0A — Repo & Operational Foundation
**Spec section:** [Helm_T1_Launch_Spec_V2.md §T0.A12](../Helm_T1_Launch_Spec_V2.md)
**Status:** Pre-build design note for architect review.

---

## What I'm building

A `.github/workflows/build-publish.yml` that builds the runtime Docker image on every PR (catches Dockerfile breakage before merge) and pushes to GitHub Container Registry on every merge to `main` (tagged `latest`, branch name, short SHA). BuildKit cache via GitHub Actions cache backend. Trivy vulnerability scan on the merged image — non-blocking, surfaced as PR comment.

## Why this shape

1. **GHCR is the substrate for Render deploys (T4.11) and preview environments (T4.6).** Render pulls from a registry; "build on Render" is slower and less reproducible. Pushing a known-good image to GHCR turns deployment into "pull this SHA" — one-line rollback.
2. **PR builds without push catch breakage early.** The cost of finding a broken Dockerfile in CI on a PR is minutes; finding it after merge is a stuck `main`. Build-on-PR is the cheapest possible safety net.
3. **`latest` + branch + SHA tags is the durable triad.** `latest` for the human "give me the current one"; SHA for the reproducible "give me the exact one"; branch for `main` recovery if something breaks. Anything less leaves a hole.
4. **Trivy non-blocking is honest.** Helm has no security team; making vuln scans block merges would either become noise (immediate dismissal) or block all work (fail-closed on every CVE in a slim base image). Surface as PR comment so the info exists without the cost.

## Alternatives considered + rejected

- **Docker Hub instead of GHCR.** Rejected — Docker Hub free tier is rate-limited on pulls. Render free tier pulling images on every restart will blow through that. GHCR has GitHub's quota, integrated with the repo.
- **Build on every commit (not just PR/merge).** Rejected — wastes CI minutes; PR + merge covers the meaningful events.
- **Multi-arch builds (amd64 + arm64).** Rejected at T1 — Render free tier is amd64; Maxwell's dev box is also amd64. Adding arm64 doubles build time for zero current consumer. Re-evaluate if anything on ARM enters the picture.
- **Cosign / signed images.** Rejected at T1 — disproportionate for a single-dev project; revisit at Stage 2.
- **Build the helm-ui in this same workflow.** Rejected — UI ships to Vercel, separate substrate; co-locating builds blurs the boundary.

## Interfaces / contracts

- **Depends on T0.A7 (Deployment Hardening)** — the Dockerfile T0.A12 builds is the one T0.A7 produced. If T0.A7 ships a broken Dockerfile, T0.A12 immediately surfaces it.
- **Provides substrate for T4.4 (Dev Deployment Decision)** — having images in GHCR makes "deploy to X" a registry-pull conversation, not a build conversation.
- **Provides substrate for T4.11 (Persistent Dev Deployment)** — Render config will reference `ghcr.io/mconn0330-svg/hammerfall-solutions/helm-runtime:latest` (or pinned SHA).
- **Provides substrate for T4.6 (Preview Environments)** — preview env per PR pulls the PR-built image (PR builds become push-on-PR for preview consumers, not push-on-merge).
- **Consumes `GITHUB_TOKEN`** with `packages: write` permission — no extra secret rotation.

## Risks + open questions for architect

- **Cache poisoning concern with `cache-to: type=gha,mode=max`.** GHA cache is per-repo; not a cross-tenant risk. But if a malicious dep slips into a build, the cache could persist the malicious layer across subsequent builds. Trivy scan should catch most; worth a flag.
- **PR builds don't push — but T4.6 (preview env per PR) needs a per-PR image to deploy.** Resolution path: T4.6's design note will reconcile this — either flip PR builds to push-with-tag like `pr-123`, or have T4.6 build its own image. I lean toward flipping push-on-PR with a `pr-NNN` tag, cleaned up on PR close.
- **`paths:` filter on `services/helm-runtime/**` means workflow doesn't run when only docs change.** Correct — but if `requirements.txt` or `Dockerfile` move out of that path during refactor, builds silently stop. Worth a "if you move these files, update the workflow filter" note in the runbook.
- **Image visibility starts public.** Spec says "flip to private in repo settings if Helm code becomes sensitive." Architect: would you rather start private and flip public if/when needed? Public default is more "show your work"; private default is more "default-deny." I lean public for T1 transparency.
- **No image size budget.** Should we enforce one (e.g., fail CI if image > 500MB)? Pro: prevents accidental bloat; Con: another budget to maintain. Ask architect whether to add at T1 or wait until image bloat is a real problem.

## What "done" looks like at the gate

- A merge to `main` triggers a build, pushes `ghcr.io/mconn0330-svg/hammerfall-solutions/helm-runtime:{latest, main, <sha>}` — verifiable via `docker pull`.
- A PR with a Dockerfile syntax error fails CI before merge.
- Subsequent merges complete in <2 minutes wall-clock thanks to BuildKit cache.
- Trivy report visible as PR comment, doesn't block merge.
- Pull access works from a clean machine without auth (assuming public visibility).
- A SITREP at PR open notes any deviations from this design note.
