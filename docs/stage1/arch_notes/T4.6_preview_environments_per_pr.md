# T4.6 — Preview Environments per PR

**Tier:** ARCH
**Phase:** 4 — Operational Readiness
**Spec section:** [Helm_T1_Launch_Spec_V2.md §T4.6](../Helm_T1_Launch_Spec_V2.md)
**Status:** Pre-build design note for architect review.

---

## What I'm building

Per-PR ephemeral runtime + UI so Maxwell can talk to changed-Helm before merging. Vercel auto-provisions the UI preview (free, native). Render service created via API on PR open, destroyed on PR close. Database isolation via Supabase project-column partitioning (`project='hammerfall-pr-<num>'`) — no Supabase branching (paid feature). Cleanup deletes partition rows on PR close.

## Why this shape

1. **Helm changes are unevaluable from a diff.** A prompt edit, a Routine change, a directive parser tweak — the only way to know if the change is good is to talk to it. T4.6 makes that talkable in <5 min from PR open.
2. **Reuse T4.11's stack — adds zero new deployment dependencies.** Same Vercel, same Render, same Supabase. T4.6 is "T4.11 with PR-scoped lifecycle." That's why T4.11 must land first.
3. **Project-column partition is the only free path.** Supabase branching is a Pro feature ($25/mo). Project-column partition is one column filter — works on free tier, isolation is at query level, cleanup is one DELETE.
4. **Per-table partition policy is a real architectural call.** Some tables are Helm's identity (`helm_personality`, `helm_beliefs`) — these stay shared across PRs (each PR talks to "the real Helm"). Other tables are conversational (`helm_memory`, `helm_frames`, `helm_messages`) — these partition (each PR has its own conversation). Without explicit policy, partition behavior would be ambiguous.
5. **Vercel previews are native — zero workflow code.** Use what's already free and good. Don't reinvent.

## Alternatives considered + rejected

- **No preview environments — review from diff + run locally.** Rejected — defeats the whole "talk to it before merging" value. Local-only review means the architect can't try the change without checking out the branch.
- **Supabase branching (paid).** Rejected at T1 — $25/mo is real money for a free-tier project; project-column partition gets us 90% of the value for $0.
- **Spin up a fresh Supabase project per PR (free tier).** Rejected — Supabase free tier limits projects per org; would hit the cap with 2-3 open PRs. Also no automated cleanup API.
- **Docker Compose locally per PR.** Rejected — defeats "talk from any device" benefit; architect can't try preview from his phone.
- **Vercel + Render only, no Supabase isolation (everyone writes to the same prod data).** Rejected — preview Helm writing into production brain is exactly the data-corruption path we don't want.
- **Tear down preview after N hours of inactivity (vs. on PR close).** Considered — Render free tier already idles services after 15 min, so the resource cost is near-zero while idle. PR-close teardown is sufficient and simpler.

## Interfaces / contracts

- **Depends on T4.11 (Persistent Dev Deployment)** — same stack, must exist first.
- **Depends on T0.A12 (CI Container Build + GHCR Publish)** — needs PR-built images. T0.A12's design note flagged that PR builds may need to push with `pr-NNN` tag for T4.6's consumption. Resolution: add push-on-PR with `pr-NNN` tag in T0.A12 (or in T4.6 if T0.A12 already shipped). Cleaned up on PR close.
- **Depends on Supabase service key + Render API key as GHA secrets** — `RENDER_API_KEY` and `SUPABASE_BRAIN_SERVICE_KEY` must be set before the workflow runs.
- **Provides talk-to-it review for ARCH-tier PRs** — the architect's review of any ARCH-tier task can include "I tried it on the preview env and saw X."
- **Provides substrate for T2.9's optional non-mocked path** — the simulation harness in real-Helm mode could run against a preview env if needed.

## Risks + open questions for architect

- **Render free tier cold start applies to previews.** First request after idle = ~30s wait. Acceptable for review (architect knows it's a preview), but worth setting expectations in the PR comment that links to the URL. Architect: include a "be patient on first hit" note in the auto-comment?
- **750-hr Render budget is shared across T4.11 (720) + T4.6 (~30/PR).** Spec calls this comfortably within budget for single-dev sequential. If two PRs are open simultaneously and each is poked frequently, we could overshoot. Mitigation: PR-close teardown is reliable; warmup cron does NOT cover preview envs (only T4.11). Architect: acceptable monitoring approach, or want explicit budget alarm?
- **Supabase cleanup is best-effort.** If PR is merged-without-close (e.g., closed without merge, or deleted branch), the workflow's `closed` action still fires. But if the workflow itself fails, partition data persists. Mitigation: weekly cleanup job that DELETEs partitions for closed PRs (small, can be added now or queued for Stage 2). Architect: include in T4.6 or defer?
- **Per-table partition policy.** I'll enumerate explicitly in the PR — currently leaning:
  - **Shared (no partition):** `helm_personality`, `helm_prompt`, `helm_beliefs`
  - **Partitioned (`project=hammerfall-pr-<num>`):** `helm_memory`, `helm_frames`, `helm_messages`, `helm_signals`, `helm_curiosities`, `helm_promises`, `helm_entities`, `helm_entity_relationships`
  - Architect: any of the partitioned ones you think should be shared? `helm_entities` is the most ambiguous — Sarah-the-friend in PR #1 is the same Sarah in PR #2, but partitioning keeps PR experiments isolated. I lean partitioned.
- **No auth on preview env URLs.** Anyone with the `helm-pr-<num>.onrender.com` URL can hit it. T0.A8's bearer token applies, but the token is shared. Architect: is shared token across all envs OK, or do you want per-PR rotated tokens? (Per-PR rotated is more work for marginal protection in a single-dev project.)
- **Preview tear-down doesn't clean GHCR `pr-NNN` images.** Image accumulation will grow GHCR storage. Mitigation: GHCR retention policy (keep last N untagged, delete `pr-*` after PR close). Architect: include in T4.6 PR or defer to image-budget cleanup task?

## What "done" looks like at the gate

- Open a PR → within ~5 min, PR comment auto-posted with two URLs (Vercel UI + Render runtime).
- `curl https://helm-pr-<num>.onrender.com/health` returns 200 (after cold start).
- UI loads, Maxwell sends a message, Helm responds — the response is from this PR's code.
- Memory writes go to `project='hammerfall-pr-<num>'`; query the production project, see no PR-tagged rows.
- Close PR → within ~2 min, Render service destroyed (verifiable in dashboard); partition rows deleted (verifiable in Supabase).
- A second PR opened simultaneously gets a different URL pair — no conflicts.
- Per-table partition policy enumerated in PR description.
- A SITREP at PR open notes any deviations from this design note.
