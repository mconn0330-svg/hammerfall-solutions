# T2.9 — Agent Simulation Test Harness

**Tier:** ARCH
**Phase:** 2 — Backend Build
**Spec section:** [Helm_T1_Launch_Spec_V2.md §T2.9](../Helm_T1_Launch_Spec_V2.md)
**Status:** Pre-build design note for architect review.

---

## What I'm building

A pytest-driven test harness that drives the runtime through canned conversations and asserts behavior — SSE events emitted, memory writes performed, response shape, multi-session continuity. Provider chain is mocked end-to-end (no real claude-sdk / local / anthropic-api calls). 13 initial scenarios: 10 within-session, 3 multi-session (curiosity carryover, promise follow-through, entity-by-alias). Runs in CI as part of the existing pytest job; passing simulations becomes a merge gate.

## Why this shape

1. **Helm's behavior is the product.** Unit tests catch broken functions; behavioral tests catch broken Helm. Without T2.9, every prompt edit or Routine 4 tweak is a behavioral regression risk that only surfaces when Maxwell notices it days later.
2. **YAML scenario files make the contract human-readable.** A scenario is "give Helm this turn, expect these events and writes." That format is also the bug-triage discipline: "Helm did X weird thing" → write a scenario reproducing it → fix lands when scenario passes. The format does double-duty as test and as bug repro.
3. **Mocked provider chain is the only way this stays free + deterministic.** Hitting real LLMs in CI is slow, expensive, and non-deterministic. Mocking at the provider boundary keeps the rest of the runtime real (memory, SSE, parser, directives).
4. **Multi-session continuity scenarios are required for ambient proof.** Within-session correctness is necessary but not sufficient. The whole point of Tier 2 brain types (T0.B7) is cross-session memory; if T2.9 doesn't exercise it, T0.B7's value is unverified.
5. **Asserting which provider in the chain was selected catches smart-routing regressions.** Provider chain abstraction (ADR-010, ADR-011) is load-bearing; a silent failover from `local` to `anthropic-api` is a $$ regression. Tests assert the expected provider responded.

## Alternatives considered + rejected

- **Real LLM calls in CI (with fixture seeds).** Rejected — non-deterministic, expensive, slow, and fragile to model updates. Would also make CI fail on Anthropic outages.
- **Snapshot tests of full SSE streams.** Considered — would catch any change. Rejected because every SSE timing tweak would re-snap; structural assertions on event types/order are tighter and more meaningful.
- **Hand-write Python test functions for each scenario.** Rejected — the YAML format keeps the bar low for adding scenarios (copy a file, edit it). Python-test scenarios mean every new behavior assertion is a 30-line PR; YAML is a 5-line PR.
- **Run the harness against a real local Helm runtime in dev (not CI).** That's separate (interactive `pytest -k 'simulation' --no-mock`); the CI variant uses mocks for determinism. Both paths exist.
- **Generate scenarios from production traces.** Considered — too heavy for T1; revisit at Stage 2 once production traffic exists.

## Interfaces / contracts

- **Depends on T0.B7 (Tier 2 Brain Types)** for scenarios 11–13 — curiosity, promise, and entity-typed scenarios require those types to exist.
- **Depends on T0.B1 (Memory Module Core)** mock surface — the in-memory Supabase mock substitutes for the real client.
- **Depends on the provider chain (ADR-010, ADR-011)** being abstracted enough that mock-mode is a config flip per provider.
- **Provides regression net for every prompt change after T0.B6.** Once T2.9 lands, "I tweaked the prompt" PRs are gated on simulations passing.
- **Consumed by T3.5 (laptop hello-world)** — multi-session scenarios formalize what the "Helm cares" 3-of-5 contract is testing for in automated form.
- **Consumed by T4.5 (T1 close SITREP)** — simulation results are part of the close report.
- **Output: `services/helm-runtime/tests/simulations/*.yaml`** — long-lived test corpus, scoped to a single directory.

## Risks + open questions for architect

- **The 13 initial scenarios may not be the right 13.** I'll start with the spec's list but expect to drop or merge some during build. Architect: do you want to review the final list before merge, or trust the build to converge?
- **In-memory Supabase mock is a fidelity risk.** Real Supabase has quirks (RLS, triggers, RPC). Mock should be lean — just CRUD + select with filters. RLS-related behavior gets a separate (slower) integration test that hits a real Supabase test schema. Architect: agree with this split, or want full RLS in the mock?
- **Multi-session scenarios use "seeded brain fixture" — what's the seeding mechanism?** Plan: a `fixtures/<scenario>/seed.sql` runs against the in-memory mock before the scenario starts. Simple, reproducible. Architect: any concern about seed drift as schema evolves?
- **Mock-mode for `claude-sdk` provider is unusual** — the SDK is itself a wrapper. Plan: mock at the HTTP layer (intercepts what the SDK sends), not at the SDK API. Keeps the mock realistic across SDK version bumps. Architect: agree?
- **CI runtime budget.** 13 scenarios × small overhead ≈ <30s. Adding 50 scenarios over time → maybe 2 min. Acceptable. If it grows >5min, parallelize. Worth flagging as a "watch this number."
- **Failure messages.** When a scenario fails, the diff between expected and actual events needs to be readable. Plan: pytest assertion helper that emits a diff like a structured-log diff. Architect: any preference for a specific format?
- **Scenario coverage of T0.B7 abstraction.** Scenarios 11–13 cover behavior; do we also want a "T0.B7 abstraction stayed intact" meta-test that asserts the module API hasn't grown unexpectedly? Probably overkill for T1; flag for Stage 2.

## What "done" looks like at the gate

- 13 scenarios in `services/helm-runtime/tests/simulations/`, all green in CI.
- Adding a 14th scenario is a 5-line YAML PR — proves the format scales.
- Provider mock can simulate: claude-sdk success, claude-sdk failure with anthropic-api fallback, local Thor reachable, local Thor unreachable.
- Multi-session scenarios (11–13) load seed fixture, run session 2, assert continuity.
- Pytest job in CI fails if any simulation fails; runs in <60s.
- Scenario authoring guide added to `docs/runbooks/` so future bug repros land here.
- A SITREP at PR open notes any deviations from this design note.

## Architect review — 2026-04-24

**Verdict:** APPROVE.

**Resolved open questions:**

- **CI runtime budget:** <30s for 13 scenarios is fine. Flag if it exceeds 5 minutes.
