## What I'm building

Three Tier 2 brain types added to Helm's brain — each its own sub-PR landed in this order:
- **T0.B7a:** `helm_entities` deepening (entity_type, aliases, attributes, salience, plus new `helm_entity_relationships` table). Schema enrichment, no new write surface beyond enriched columns.
- **T0.B7b:** `helm_curiosities` (open questions Helm holds across sessions).
- **T0.B7c:** `helm_promises` (commitments Helm tracks and follows up on).

Each sub-PR adds ≤ ~80 lines to the memory module proper. STOP gate after the third PR. Pulled into T1 (was Finding #001) because the cost-of-return after T0.B6 was higher than the cost of doing it now, and because T3.5 hello-world is materially better with curiosity carryover and promise discipline.

## Why this shape

1. **T0.B7 is also the abstraction validation for T0.B1.** Spec is explicit: if any sub-PR exceeds ~150 lines in the module proper, T0.B1's abstraction failed and gets revisited before continuing. This is a contract, not a hope. The 50-line target makes that test sharp.
2. **Order matters: deepening → first new type → second new type.** T0.B7a (deepening) is the lowest-risk template since it's pure ALTER on an existing table. T0.B7b is the first time the abstraction is actually tested with a brand-new type. T0.B7c is the proof the pattern holds — should feel like copy-paste-modify of T0.B7b.
3. **One type per PR, not all three together.** Each PR is reviewable on its own; if T0.B7b reveals an abstraction problem, we stop and fix T0.B1 before T0.B7c instead of unwinding three types at once.
4. **Promise-detection lives in the prompt, not the module.** "Was that utterance a promise?" is fuzzy and judgmental — exactly the kind of thing prompts handle and modules shouldn't. Module stays unopinionated; prompt owns the heuristic.
5. **Salience decay default = 1.0 (no decay).** Decay logic is post-T1 work; the column exists now so Stage 2 can implement decay without a migration, but T1 doesn't actively decay anything. Same shape applies to confidence on relationships.

## Alternatives considered + rejected

- **Land all three in one PR.** Rejected — collapses three review surfaces into one; the abstraction-validation signal is also lost.
- **Land Tier 2 post-T1 (per original Finding #001).** Was the plan; reversed because cost-of-return was higher than cost-of-doing-now (memory module + agent prompts both fresh) and because hello-world depth materially improves with these types live.
- **Skip `helm_entity_relationships` at T0.B7a, defer to Stage 2.** Considered — but the relationship table is the smallest possible "we'll need it for Tier 3" demonstration, and adding it now means Tier 3 (helm_anticipations, helm_tensions) doesn't have to retroactively rewire entity-to-entity edges.
- **Make `helm_curiosities` and `helm_promises` a single "intentions" table with a `kind` column.** Considered — but the lifecycles are different (curiosities resolve, promises fulfill; curiosity status is open/investigating/resolved/abandoned, promise status is open/fulfilled/missed). Joining them creates more conditional logic than two clean tables.
- **Add curiosity/promise *detection* to the module.** Rejected — see why-this-shape #4.

## Risks + open questions for architect

- **The "150-line threshold" is somewhat arbitrary.** It's a heuristic, not a measured number. Architect: do you want a different threshold, or accept this as the "smell test" floor?
- **T0.B7a is back-compat-required (existing entities mid-migration).** `write_entity()` must accept old (no type) and new (with type) signatures. Plan: type defaults to NULL on legacy rows; new writes always set type. Backfill of legacy rows is post-T1 cleanup, not blocking. Architect: fair?
- **Prompts are growing in this PR.** Routine 4 in `helm_prompt.md` gets new sections for curiosity-formation, promise-formation, session-end promise-fulfillment check. Concern: prompt length affects every turn's TTFT. Mitigation: T4.8 SLO bench will catch regressions; if TTFT degrades, the additions move to a smaller "active reminders" prompt loaded only when relevant. Architect: pre-emptive split, or wait for the bench signal?
- **`Helm_Brain_Object_Types.md` doc reorganization on each sub-PR.** Three doc updates inside one ARCH task is a lot of doc churn. Acceptable since each sub-PR represents real movement, but worth flagging.
- **No migration-down logic specified.** ALTER ADD COLUMN is reversible in principle; the spec doesn't require a tested down-migration. Architect: do you want each migration paired with a tested rollback, or accept "forward only" since we have backups (T0.A10)?
- **Test isolation between sub-PRs.** T0.B7b and T0.B7c add to the same `tests/test_memory.py` (or sibling files). If T0.B7b's tests assume a clean DB and T0.B7c's also do, any shared fixture work needs coordination. Plan: per-type test file, per-type fixture. Architect: agree?

## What "done" looks like at the gate (per sub-PR)

**T0.B7a:**
- Migration applies to a copy of production schema; reverses cleanly (manual test).
- `write_entity(name="Sarah", entity_type="person", aliases=["Sarah Chen", "Chen"])` round-trips.
- `helm_entity_relationships` row links two entities; cascade delete works.
- `read_entities(entity_type="person")` returns only persons.

**T0.B7b:**
- `write_curiosity(text="Why does X happen?")` → `read_open_curiosities(project)` lists it.
- `update_curiosity_status(id, "resolved", resolution="...")` → drops out of opens.
- Prime context loader includes top-5 open curiosities by priority.
- Module diff in this PR ≤ ~150 lines (pass = 50–80; fail-flag = 150+).

**T0.B7c:**
- `write_promise(text="I'll watch for over-engineering")` → `read_open_promises(project)` lists it.
- `update_promise_status(id, "fulfilled", fulfillment_entry_id=...)` round-trips.
- Prime context loader surfaces with relative due-time formatting.
- Module diff in this PR ≤ ~150 lines AND structurally similar to T0.B7b.

**STOP gate (post-third-PR):** Maxwell signs off that laptop hello-world will surface curiosity + promise context.

## Architect review — 2026-04-24

**Verdict:** APPROVE.

**Resolved open questions:**
- **150-line threshold:** Accepted as smell test. If T0.B7b exceeds it, stop and revisit T0.B1 before T0.B7c.
- **T0.B7a back-compat:** `entity_type` defaults to NULL on legacy rows; new writes always set it. Backfill is post-T1 cleanup, not blocking.
- **Prompt growth concern:** No preemptive split. Wait for T4.8 bench data; address with measurement, not anticipation.
- **No migration-down logic:** Forward-only is fine given T0.A10 backups. ALTER ADD COLUMN reverses trivially in principle.
- **Per-type test file, per-type fixture:** Confirmed as the correct isolation pattern.

**Confirmed dependency chain:** T0.B6 (prompt rewrite, position 21 in execution order) must be complete before T0.B7 (position 22). Sequencing in the spec is respected.
